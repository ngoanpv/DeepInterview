# Supabase — DeepInterview

This directory holds the database schema for DeepInterview. Postgres + Auth +
Storage are provided by Supabase; the agent writes interview state to two tables.

## Tables

### `public.profiles`
One row per authenticated user (auto-created on sign-up by the
`on_auth_user_created` trigger → `handle_new_user()`).

| column            | type          | notes                                  |
| ----------------- | ------------- | -------------------------------------- |
| `id`              | uuid (PK)     | FK → `auth.users(id)`, cascade delete  |
| `email`           | text          | copied from the auth user              |
| `plan`            | text          | billing tier, default `free`           |
| `interviews_used` | integer       | per-tier interview cap counter         |
| `locale`          | text          | preferred UI language, default `en`    |
| `created_at`      | timestamptz   | default `now()`                        |

### `public.sessions`
One row per interview session. The agent's `SupabaseRepository` maps to these
columns: it generates a text `id` (`sess_<uuid>`), stores the full
`InterviewContext.model_dump()` in `context`, the `ScoreCard.model_dump()` in
`scorecard`, and the live transcript turns in `transcript`.

| column          | type        | notes                                            |
| --------------- | ----------- | ------------------------------------------------ |
| `id`            | text (PK)   | `sess_<uuid>` generated by the agent             |
| `user_id`       | uuid        | FK → `public.profiles(id)`, nullable             |
| `status`        | text        | `prep` → `ready` → `complete`, default `prep`    |
| `company`       | text        | from `PrepRequest.company`                        |
| `cv_url`        | text        | from `PrepRequest.cv_url`                          |
| `jd_text`       | text        | from `PrepRequest.jd_text`                         |
| `language_mode` | jsonb       | `{"primary":"en","mixed":false}`                  |
| `context`       | jsonb       | serialized `InterviewContext`                     |
| `scorecard`     | jsonb       | serialized `ScoreCard`                            |
| `transcript`    | jsonb       | list of turn objects                              |
| `created_at`    | timestamptz | default `now()`                                   |
| `updated_at`    | timestamptz | bumped on every update by the `sessions_touch` trigger |

## Applying the migration

**Supabase CLI (recommended):**

```bash
supabase link --project-ref <your-project-ref>
supabase db push          # applies supabase/migrations/*.sql
```

**Or paste into the SQL editor:** open the Supabase dashboard → SQL Editor → new
query → paste the contents of `migrations/0001_init.sql` → Run.

## Row Level Security

RLS is **enabled** on both tables. The policies are owner-scoped:

- `own profile` — a user can read/write only the `profiles` row where
  `auth.uid() = id`.
- `own sessions` — a user can read/write only `sessions` rows where
  `auth.uid() = user_id`.

This protects the browser (anon/auth) client. The **agent** does not go through
RLS: it uses the **service-role key** (`SUPABASE_SERVICE_ROLE_KEY`), which
bypasses RLS, so it can create sessions with `user_id` set server-side and update
status/context/scorecard regardless of the policies. Keep the service-role key
server-only — never expose it to the browser.

## Agent configuration

Set these env vars for the agent to use Supabase instead of the in-memory store:

```
SUPABASE_URL=https://<project-ref>.supabase.co
SUPABASE_SERVICE_ROLE_KEY=<service-role-key>
```

With both present, `get_repository()` returns the `SupabaseRepository`; otherwise
it falls back to the process-local `MemoryRepository`.
